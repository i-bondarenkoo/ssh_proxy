Сервис для подключения к unifi-точкам по ssh 
Реализован на основе 2 классов.
/app/services
UnifiDevice - класс отвечает за создание инстансов конкретных точек в сети. У объекта создается 3 атрибута - ip адрес, лок для последовательного выполнения запросов к одной точке и connect для сохранения ssh-соединения. Реализованы 3 основных метода.
-get_ssh_session - отвечает за устновку ssh соединения с точкой. Установлены 2 не стандартных параметра connect_timeout 1 секунда, keepalive_interval=30 отправляет каждые 30 секунд специальные пакеты/пинги на точку, для проверки активно ли соединение
-run_coomand и декоратор для подсчета времени. В цикле из 2 итераций пытаемся установить соединение с точкой и выполнить команду. Возвращает словарь из 3 ключей -
{
                        "stdout": result.stdout,
                        "stderr": result.stderr,
                        "exit_status": result.exit_status,
} вывод команды, ошибка если есть, статус выполнения. Так же обрабатывает 2 возможных ошибки - asyncssh.Error (возникает при проблемах с установкой соединения) всего попыток - 2. И OSError возвникает в случае обращения к адресу, которого нет в сети
-close используется для корректного закрытия все соединений при закрытии приложения

SSHManager - класс который сохраняет данные о конкретных точках. Данные хранятся в словаре, где dict[str, UnifiDevice] ключ это строка(ip адрес точки), значений - объект UnifiDevice. Методы run и close_all_ssh_session вызываются непосредственно у каждого объекта, так как менеджер ничего не знает о конкретных точках, он только выполняет команды.

/app/core/config
Небольшой класс, который умеет читать значения из перменных окружения, сами данные для подключения username/password задаются в .env файле и интерпретатор во время выполнения кода берет эти данные из оперативной памяти операционной системы

/app/core/logging
Логер для отладки и сбора статистики. Базовый уровень для отображения в консоль/файл - INFO. Данные отобрадаются в формате filename/levelname/message. Так же есть запись логов в файл /opt/log/app.log (путь до файла на сервере)
/app/views/proxy

Эндпоинт типа post /execute принимает данные в формате тело-запроса, это json с полями {ip: str, command: str} (при отправке не корректных данных вылезет ошибка 422)

Работа с самим ssh-менеджером реализована через жизненный цикл приложения. При старте инициализируется 1 инстанс менеджера на все приложение в котором будут храниться адреса точек и активные подключения. При закрытии приложения все ssh сессии так же корректно закрываются

Для запуска.
-создать .env файл в папке app 
-собрать образ docker build -t ssh-proxy
-запустить контейнер - docker run -d --name ssh-proxy -p 8000:8000 --env-file .env -v /opt/log:/opt/log ssh-proxy

Пример запроса и ответа.
curl -X POST http://localhost:8000/execute -H "Content-Type: application/json" -d '{"ip":"192.168.15.2","command":"uname -a"}'
